# V3 — "La plateforme de données du développeur"

> **Objectif** : Transformer QoreDB d'un client de base de données en le centre de gravité du workflow data du développeur — investigation, documentation, qualité, collaboration.

---

## 📓 Database Notebooks (Feature majeure)

- [x] **Notebook MVP** : Document exécutable mêlant cellules SQL/Mongo et Markdown, connecté à une base live. (Spec complète : `NOTEBOOK_SPEC.md`)
  - [x] Backend : commandes `save_notebook`, `load_notebook`, `list_notebooks`
  - [x] Frontend : `NotebookTab`, `NotebookCell`, `SqlCell`, `MarkdownCell`, `CellResultViewer`
  - [x] Nouveau type de tab `'notebook'` dans le système d'onglets
  - [x] Format fichier `.qnb` (JSON, diffable avec Git)
  - [x] `Ctrl+Enter` pour exécuter, `Ctrl+S` pour sauvegarder, drag-and-drop pour réordonner

- [x] **Variables paramétrées** : Barre de variables typées (`$customer_id`, `{{date_from}}`) avec inputs automatiques (texte, nombre, date, select).
- [x] **Run All / Run From Here** : Exécution séquentielle des cellules avec arrêt sur erreur (configurable).
- [x] **MongoCell** : Support des cellules MongoDB dans les notebooks.
- [x] **Indicateurs visuels d'état** : Bordures colorées (idle/running/success/error/stale) pour chaque cellule.
- [x] **Import depuis .sql / .md** : Conversion automatique de fichiers existants en notebooks.
- [x] **Export Markdown** : Génération d'un `.md` lisible (GitHub, Notion) avec code blocks SQL et tables de résultats.
- [x] **Export HTML standalone** : Fichier `.html` autosuffisant avec syntax highlighting et résultats stylés.
- [x] **Références inter-cellules** [Pro] : Référencer le résultat d'une cellule dans une autre (`$cell_label.column`).
- [x] **Cellules Chart** [Pro] : Visualisations basiques (bar, line, pie, scatter) référençant une cellule source.
- [x] **"Convert Query to Notebook"** : Transformer un onglet query en notebook en un raccourci (`Ctrl+Shift+N`).

## ⚡ Killer Features

- [ ] **Cross-Database Federation Pro** [Pro] : Enrichissement de la fédération existante.
  - [ ] Syntaxe qualifiée `connexion.schema.table` dans l'éditeur SQL
  - [ ] Auto-matérialisation intelligente dans DuckDB (cache des sources peu changeantes)
  - [ ] Pushdown de prédicats amélioré (filtres, limites poussés vers la source)
  - [ ] Support des sources MongoDB dans les requêtes fédérées (aplatissement document → table)

- [ ] **Data Time-Travel** [Pro] : "Git blame" pour les données, basé sur l'audit trail de l'intercepteur.
  - [ ] Timeline visuelle des mutations par table (INSERT/UPDATE/DELETE capturés par QoreDB)
  - [ ] Diff entre deux points dans le temps pour une ligne donnée
  - [ ] Restauration d'un état précédent (génération du SQL de rollback)
  - [ ] Filtre par utilisateur/session et plage temporelle
  - [ ] Scope : mutations faites via QoreDB uniquement (pas de WAL/binlog)

- [ ] **Data Contracts** [Pro] : Assertions sur la qualité des données, exécutables et monitorables.
  - [ ] Définition YAML/JSON : règles par table/colonne (NOT NULL %, format regex, range, unicité, FK intégrité)
  - [ ] Exécution on-demand ou intégrée au notebook (cellule de type `contract`)
  - [ ] Dashboard de santé : pass/fail par contrat, tendances, indicateur dans la sidebar (vert/orange/rouge)
  - [ ] Intégration avec l'intercepteur : alerte quand une mutation viole un contrat actif

- [ ] **Instant Data API** [Pro] : Exposer des requêtes sauvegardées comme endpoints REST locaux.
  - [ ] Sélection d'une query dans la library → "Expose as API"
  - [ ] Micro-serveur HTTP local (port configurable, localhost uniquement par défaut)
  - [ ] Paramètres de query mappés comme query params
  - [ ] Réponse JSON avec pagination automatique
  - [ ] Documentation auto-générée (OpenAPI/Swagger basique)

- [ ] **Environment-Aware Workspaces** : Organisation des connexions par projet/environnement.
  - [ ] Répertoire `.qoredb/` par projet (connexions, notebooks, query library, data contracts, contexte IA)
  - [ ] Détection automatique au lancement si le CWD contient un `.qoredb/`
  - [ ] Switch de workspace dans la sidebar
  - [ ] Opérations cross-environnement facilitées (diff Prod vs Staging en un clic)
  - [ ] Versionnable avec Git : toute l'équipe bénéficie de la même configuration

- [ ] **Query Replay Lab** [Pro] : Enregistrer des queries, les rejouer après migration, détecter les régressions.
  - [ ] Mode "Record" : capture les queries d'une session dans un replay set via l'intercepteur
  - [ ] Mode "Replay" : ré-exécute le set sur une connexion cible, rapport des écarts
  - [ ] Comparaison visuelle des résultats avant/après migration (intégration avec le Visual Data Diff)
  - [ ] Rapport : queries cassées, résultats différents, nouvelles erreurs

## ⚙️ Performance & Stabilité

- [ ] **Connection Pooling Tuning** : Tuning des pools par driver, métriques de pool exposées dans les dev metrics.
- [ ] **Query Result Caching** : Cache local des résultats SELECT récents, invalidation sur mutation détectée via l'intercepteur.
- [x] **Row Count Optimization** : Comptage rapide pour les très grandes tables (> 10M lignes) via statistiques système (`pg_class.reltuples`, `information_schema`).
- [ ] **Memory Profiling** : Détection proactive des fuites mémoire dans les longues sessions (monitoring consommation par onglet).
- [ ] **Startup Time Optimization** : Lazy-loading des drivers, initialisation différée des services non critiques (intercepteur, snapshot store).
- [ ] **DuckDB Async Wrapper** : Migration du wrapper `spawn_blocking` actuel vers une API asynchrone pour éviter le blocage du thread pool tokio.

## 🔮 V3+ — Appendice (Recherche & Exploration)

> Ces features sont identifiées comme stratégiques mais trop complexes ou prématurées pour V3. Elles seront évaluées en fonction du feedback utilisateur et de la traction du produit.

### Plateforme

- [ ] **Schema-as-Code Bridge** [Pro] : Sync bidirectionnelle entre ORM (Prisma, Drizzle, TypeORM) et DB réelle.
  - Drift detection entre schéma ORM, DB dev, DB staging, DB prod
  - Génération de migration à partir du diff
  - Parser ORM spécifique par framework (commencer par Prisma)
  - Très complexe — évaluer si un partenariat ou plugin externe est plus réaliste

- [ ] **Query Replay Lab (Avancé)** [Pro] : Version complète du replay lab.
  - Analyse sémantique des résultats (au-delà du diff texte)
  - Scoring de régression automatique
  - Intégration CI/CD (replay en pipeline, rapport dans les PR)

- [ ] **Self-Host Sync** [Team] : Synchronisation multi-device via serveur auto-hébergé.
  - Connexions, notebooks, query library, data contracts
  - Conflict resolution
  - Chiffrement end-to-end

- [ ] **Real-Time Collaboration** [Team] : Notebooks collaboratifs avec édition simultanée.
  - CRDT pour la résolution de conflits
  - Présence (curseurs, sélection)
  - Nécessite une infrastructure serveur

### Drivers additionnels

- [ ] **Neo4j** : Graph database. UI de visualisation de graphes spécifique. Niche mais différenciant.
- [ ] **Cassandra / ScyllaDB** : Wide-column stores. CQL proche de SQL — potentiellement adaptable.
- [ ] **Oracle** : Demande enterprise forte. Pilote natif complexe (OCI).
- [ ] **Firestore** : Mode document (mentionné dans PROJECT.md). SDK Google.
- [ ] **InfluxDB / QuestDB** : Bases time-series pures (complémentaire à TimescaleDB).

### Enterprise

- [ ] **SSO / SAML** [Enterprise] : Authentification centralisée.
- [ ] **Role-Based Access Control** [Enterprise] : Permissions fines par utilisateur/rôle.
- [ ] **Policy Configuration** [Enterprise] : Verrouillage de paramètres de sécurité par l'administrateur.
- [ ] **Audit Logs (équipe)** [Enterprise] : Trail d'audit centralisé partagé par l'équipe.
- [ ] **Plugin Marketplace** : Distribution et découverte de plugins communautaires.
